WebKit / Web Audio

Why it takes three moves

The gesture

Start on a gesture that completes — pointerup, touchend, click — never pointerdown. iOS treats a pointerdown as a maybe-scroll and withholds audio permission until the gesture ends, so resume() quietly does nothing.

The retry

One resume() often leaves it suspended. Loop until the state is running, and re-resume whenever iOS drops the session to interrupted for a call, Siri, or a route change.

The session

Set navigator.audioSession.type = 'playback' (iOS 16.4+) so output rides the media channel and the hardware ringer switch stops muting it.

The whole thing

ios-web-audio.js — usage 
// The 20-line core lives in ios-web-audio.js. Usage is three lines:
import { makeUnlockableAudioContext } from './ios-web-audio.js';

const audio = makeUnlockableAudioContext();

// pointerup, NOT pointerdown — the gesture must complete
button.addEventListener('pointerup', async () => {
  const running = await audio.unlock();  // session + retried resume()
  // build your graph on audio.ctx and start playing
});
